# I3C Dual-Bus SoC Educational Simulator

Pure Python MIPI I3C SDR protocol model + rich interactive HTML5 visualizer.

## What it demonstrates

- One host SoC containing **two independent I3C controllers** (Bus 0 + Bus 1)
- Full bus initialization using **Dynamic Address Assignment (DAA)**
- Broadcast and Direct **Common Command Codes (CCC)**
- **Private Write / Private Read** transfers (sensor config + data acquisition)
- **In-Band Interrupt (IBI)** from an I3C target
- **Host-mediated cross-bus communication** (data read on one bus is forwarded as a command to a device on the second bus)
- Mixed I3C + legacy I²C style devices on the same bus

## Architecture

```
Python Model (i3c_model.py)
├── I3CBus          – signal event generator + transaction log
├── I3CController   – protocol engine (START/STOP, addressing, DAA, CCC, private xfers, IBI)
└── build_soc_scenario() → rich JSON trace (waveforms + typed transactions)

HTTP Server (server.py, stdlib only)
└── Serves beautiful self-contained HTML/JS visualizer
    ├── Two synchronized waveform canvases (SCL + SDA)
    ├── Clickable transaction table with live filtering
    ├── Detailed Korean/English interpretation for every transaction
    ├── Playhead scrubber + real-time animation
    └── Re-simulate button (new random sensor values each time)
```

## Run

```bash
cd i3c-simulator
python3 server.py
```

Browser will open automatically at http://127.0.0.1:8088

## Keyboard Shortcuts (in browser)

| Key          | Action                    |
|--------------|---------------------------|
| `Space` / `K`| Play / Pause animation    |
| `←` `→`      | Seek ±800 ticks           |
| `R`          | Reset to t=0              |
| `F`          | Cycle bus filter          |
| Click waveform | Seek to that time       |
| Click table row | Jump + show interpretation |

## Re-generate trace

Click **재시뮬레이션** button in the UI — each run produces slightly different sensor data while keeping the same protocol flow.

## Educational value

Every transaction carries a human-readable `interpretation` field generated by the Python model. This is the key teaching tool:

- What a particular CCC actually does
- Why DAA arbitration order matters
- What an IBI means in a real sensor hub
- How a host typically moves data between two separate I3C domains inside one chip

## Files

- `i3c_model.py` — complete standalone protocol model (no external deps)
- `server.py`   — zero-dependency HTTP server + embedded HTML/JS UI
- `README.md`   — this file

## Limitations (by design)

- Simplified DAA (no full 48-bit PID + BCR + DCR + T-bit parity on wire, but visually and semantically faithful)
- No HDR-DDR / HDR-TSP / HDR-BT
- No full multi-master arbitration or hot-join
- Timing is abstract (good for visualization, not cycle-accurate RTL)

The goal is **protocol understanding**, not replacement of a real I3C VIP or logic analyzer.
